# 12.2. Código fuente (Repositorios)

## 12.2.1. Repositorio de base de datos y documentación

- **Repositorio:** `bd252-grupo5`

Este repositorio contiene principalmente:

- Scripts de **definición de esquemas y tablas (DDL)**:
  - Carpeta `scripts/ddl/tables` (o equivalente), incluyendo el script consolidado `consolidado.sql` donde se define:
    - El esquema operacional `monitoreo` y tablas relacionadas al módulo.
    - El esquema analítico `monitoreo_analytics` (dimensiones y tabla de hechos).
    - Las funciones PL/pgSQL `f_cierre_diario_operaciones` y `f_cierre_rango_fechas`.
- Scripts de **poblamiento de datos de prueba**:
  - Carpeta `scripts/poblamiento_datos/codigo_python/` con el script `consolidado.py` que utiliza Faker para generar datos.
  - Carpeta `scripts/poblamiento_datos/scriptSQL/` con inserts SQL asociados.
- Documentación del proyecto:
  - Capítulos de diseño, alcance del piloto, flujos y procesos batch (por ejemplo, secciones 9.x y 11.x).

Este repositorio sirve como **fuente de verdad** de la estructura de la base de datos, de los procesos batch y de la documentación formal del módulo.

## 12.2.2. Repositorio de aplicación (backend + frontend)

- **Repositorio:** `bd252-grupo5-app`

Este repositorio contiene el código de la aplicación implementada para el piloto, dividida en:

### Backend (`backend/`)

- Framework: **NestJS** con **TypeORM**.
- Estructura relevante para el Módulo 5:
  - `backend/src/monitoreo/`
    - `entities/`: entidades de dominio (Operación, Contenedor, Notificación, Incidencia, Entrega, Importador, etc.).
    - `services/`: servicios de aplicación, incluyendo:
      - `reportes.service.ts`: lógica de generación de reportes y ejecución del batch analítico (cierre diario y cierre de rango de 120 días).
    - `controllers/`: controladores HTTP/REST, incluyendo:
      - `reportes.controller.ts`: endpoints para Gestión de Reportes y Analytics (`/monitoreo/reportes/...`).
  - Integración con PostgreSQL mediante un `DataSource` compartido.

### Frontend (`frontend/`)

- Framework: **Next.js (App Router)** con **React** y **Tailwind CSS**.
- Estructura relevante para el Módulo 5:
  - `frontend/app/monitoreo/`
    - `operaciones/`: pantallas para listar, ver y registrar operaciones.
    - `contenedores/`: pantallas de lista y detalle de contenedores, sensores y notificaciones.
    - `notificaciones/`: panel de notificaciones.
    - `incidencias/`: registro y gestión de incidencias.
    - `entregas/`: registro y gestión de entregas a importadores.
    - `reportes/`: gestión de reportes y acceso a analítica.
      - `page.tsx`: listado de reportes y bloque de Proceso Batch (cierre diario y cierre 120 días).
      - `analytics/page.tsx`: dashboard de estadísticas basado en `monitoreo_analytics`.
  - `frontend/context/AuthContext.tsx`: contexto de autenticación con JWT.

## 12.2.3. Relación entre repositorios

- El repositorio `bd252-grupo5` define la **base de datos y los procesos batch**, y proporciona scripts para crear y poblar el entorno.
- El repositorio `bd252-grupo5-app` implementa la **lógica de negocio y las interfaces de usuario** que consumen esa base de datos:
  - El backend usa las tablas y funciones definidas en `consolidado.sql`.
  - El frontend consume los endpoints expuestos por el backend para operar sobre los datos operacionales y analíticos.
- Para el piloto de 120 días:
  - Se ejecutan primero los scripts de poblamiento (Python + SQL) del repositorio `bd252-grupo5`.
  - Luego se utiliza el endpoint `/monitoreo/reportes/cierre-rango-120` del backend (en `bd252-grupo5-app`) para poblar la fact analítica.
  - Finalmente, el dashboard de analítica en el frontend muestra los resultados consolidados.
