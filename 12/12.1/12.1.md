# 12.1. Alcance de la implementación

## 12.1.5 Alcance Módulo 5

### Alcance funcional implementado

Se implementó un flujo end-to-end para el monitoreo de entregas, alineado con el flujo funcional descrito en [9.2.5](/9/9.2/9.2.5/9.2.5.md):

- **Registro y gestión de operaciones de monitoreo**
  - Creación de operaciones de monitoreo con asignación de contenedores.
  - Asociación de medios de transporte (vehículos/buques) según corresponda.
  - Edición y visualización detallada de las operaciones.

- **Monitoreo en tiempo real**
  - Dashboard de monitoreo con lista de operaciones y KPIs básicos.
  - Visualización de posiciones GPS en un mapa para los contenedores/vehículos.

- **Detalle de contenedores y sensores**
  - Vista detallada de contenedor con datos maestros.
  - Listado de sensores asociados y últimas lecturas registradas.
  - Acceso al historial de lecturas y notificaciones de cada sensor.

- **Notificaciones e incidencias**
  - Panel de notificaciones generadas por los sensores (alertas, advertencias, etc.).
  - Registro, edición y visualización de incidencias asociadas a operaciones.

- **Entrega final al importador**
  - Registro de entregas de contenedores a importadores.
  - Asociación con la operación de monitoreo y el contenedor entregado.
  - Consulta y edición de entregas registradas.

- **Reportes y analítica**
  - Gestión de reportes derivados de operaciones, notificaciones e incidencias.
  - Proceso batch analítico para consolidar información en un esquema `monitoreo_analytics`.
  - Dashboard de analítica con KPIs y gráficos basados en datos históricos.

Todo este flujo está respaldado por las pantallas descritas en [12.3.5](/12/12.3/12.3.5/12.3.5.md).

## 12.2 Alcance técnico implementado

### Backend (NestJS + TypeORM + PostgreSQL)

- Implementación de entidades principales del módulo de monitoreo (operaciones, contenedores, sensores, notificaciones, incidencias, entregas, importadores, etc.).
- Exposición de endpoints REST para:
  - CRUD de operaciones, contenedores, notificaciones, incidencias, reportes y entregas.
  - Consulta de datos de monitoreo para el dashboard y el detalle de contenedor.
  - Ejecución del proceso batch diario analítico: `POST /monitoreo/reportes/cierre-diario`.
  - Ejecución del proceso batch de rango (120 días): `POST /monitoreo/reportes/cierre-rango-120`.
  - Consulta de resumen analítico: `GET /monitoreo/reportes/analytics/resumen`.
- Implementación del esquema analítico `monitoreo_analytics` y de las funciones PL/pgSQL:
  - `f_cierre_diario_operaciones(p_fecha_corte DATE)`.
  - `f_cierre_rango_fechas(p_fecha_inicio DATE, p_fecha_fin DATE)`.

### Frontend (Next.js App Router + React + Tailwind CSS)

- Implementación de pantallas de:
  - Login y perfil de usuario (integradas con el contexto de autenticación).
  - Gestión de operaciones de monitoreo y dashboard en tiempo real.
  - Detalle de contenedor con secciones de sensores, notificaciones y documentación.
  - Panel de notificaciones e incidencias.
  - Gestión de entregas e importadores.
  - Gestión de reportes y dashboard de analítica.
- Uso de componentes reutilizables (por ejemplo, `MapHeader`) y diseño consistente con el resto del sistema.

### Scripts de datos y entorno de piloto

- Scripts en Python (Faker) para poblar datos de prueba con fechas controladas mediante una constante `BASE_NOW`, permitiendo un **rango de 120 días** coherente con el piloto.
- Scripts SQL de poblamiento y DDL consolidados en el repositorio de base de datos.
- Uso del proceso batch de rango de fechas para poblar rápidamente la tabla de hechos analítica para el piloto.

## Fuera de alcance / simplificaciones

Algunos aspectos fueron deliberadamente simplificados o quedaron fuera del alcance de esta implementación:

- **Orquestación automática del batch diario**
  - El cierre diario se ejecuta de forma **manual** desde la interfaz de Gestión de Reportes.
  - No se configuró un scheduler real en producción (cron, `@nestjs/schedule`, etc.), aunque se describen opciones en la documentación (sección 11.2.5).

- **Integraciones externas reales**
  - No se integraron APIs externas de tracking GPS real ni plataformas IoT; se trabaja con datos simulados.

- **Gestión avanzada de seguridad y permisos**
  - El sistema implementa autenticación básica con JWT y contexto de usuario, pero no se desarrolló un módulo completo de autorización por roles/permisos finos.

- **Cobertura completa de todos los RF secundarios**
  - El foco estuvo en cubrir de forma sólida el flujo principal de monitoreo, incidencias, reportes y entregas. Algunos requerimientos menos críticos se documentan pero no se implementan al 100%.
