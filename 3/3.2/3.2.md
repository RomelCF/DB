# 3.2. Módulo de Gestión del Personal y Tripulacion

## Requerimientos funcionales

| **Código** | **Requerimiento Funcional** | **Caso de Uso** |
| -- | -- | -- |
| RF01 | Administrar la información de capitanes, oficiales y personal de cubierta asignados a los buques. | CU01 |
| RF02 | Registrar y controlar contratos laborales del personal administrativo y de tripulación. | CU02 |
| RF03 | Registrar cursos, certificaciones y sus fechas de vencimiento. | CU03 |
| RF04 | Administrar esquemas de trabajo y descanso de la tripulación. | CU04 |
| RF05 | Vincular tripulación con buques específicos y operaciones de embarque. | CU05 |

## Diagramas de casos de uso

### CU01: Administración de información de capitanes, oficiales y personal de cubierta

- **Actores involucrados**
  - Coordinador de Flota
  - Administrador de Recursos Humanos
  - Sistema TMS / Módulo de Gestión de Personal

- **Obejtivo**
  - Registrar, actualizar y consultar la información de capitanes, oficiales y personal de cubierta asignados a los buques, garantizando que cada embarque cuente con la tripulación adecuada.

- **Precondiciones**
  - El usuario debe estar autenticado en el sistema con permisos de gestión de personal.
  - Los buques deben estar registrados en el sistema para permitir asignaciones.

- **Disparador o evento inicial**
  - El administrador requiere de personal en un buque para que pueda zarpar.

- **Flujo principal de eventos**
    1. El tripulante accede al módulo de gestión de personal.
    2. El sistema muestra su perfil y estado actual (activo, pendiente de verificación, en embarque).
    3. El tripulante sube sus certificados de capacitación, contratos y otros documentos requeridos.
    4. El sistema valida el formato de los documentos y registra la fecha de vencimiento.
    5. El administrador accede al módulo de gestión de tripulación.
    6. El sistema muestra la lista de tripulantes con documentos pendientes de verificación.
    7. El administrador revisa que el perfil del tripulante esté completo y que todos los certificados estén vigentes.
    8. El administrador selecciona un buque y asigna al tripulante a un rol específico (capitán, oficial, personal de cubierta).

- **Flujos alternativos**
    - Si falta información obligatoria → el sistema no permite guardar hasta completar.
    - Si el buque no tiene disponibilidad de plazas para más tripulantes → el sistema muestra un mensaje de alerta.

- **Postcondiciones**
    - La información del personal queda actualizada en el sistema.
    - El tripulante queda vinculado al buque correspondiente.

- **Excepciones**
    - Registro duplicado del mismo tripulante.
    - Inconsistencia entre buque y rol asignado.

### CU02: Registro y control de contratos laborales

- **Actores involucrados**
  - Personal administrativo y Tripulación
  - Sistema TMS (Transportation Management System)

- **Obejtivo**
  - Registrar y almacenar los contratos laborales del personal de la tripulación.

- **Precondiciones**
  - El usuario debe llenar una ficha de solitud.

- **Disparador o evento inicial**
  - El administrador requiere de personal en un buque para que pueda zarpar

- **Flujo principal de eventos**
    1. El usuario accede al sistema y completa la formalidad con sus datos personales, documentación y requisitos previos.
    2. El sistema registra la formalidad y notifica al administrador que hay una solicitud pendiente de contrato o renovación.
    3. El administrador accede al módulo de gestión de personal.
    4. El sistema muestra la lista de formalidades pendientes de revisión.
    5. El administrador selecciona un trabajador y valida que la información y documentos estén completos.
    6. Si todo es correcto, el administrador genera el contrato laboral o aprueba la renovación.
    7. El sistema guarda el contrato en el historial del trabajador.
    8. El administrador selecciona un buque y asigna al tripulante a un rol específico (capitán, oficial, personal de cubierta).

- **Flujos alternativos**
    - Si el contrato ya existe para el mismo periodo → el sistema alerta duplicidad y solicita revisión.
    - Si faltan datos obligatorios → el sistema no permite registrar el contrato hasta completar la información.

- **Postcondiciones**
    - El contrato laboral queda vinculado al trabajador en su perfil.
    - El sistema actualiza el estado del trabajador como “contrato vigente”.
    - Se generan alertas para renovación en fechas próximas al vencimiento.

- **Excepciones**
    - Contrato ilegible o en formato no permitido.
    - Intento de registrar un contrato vencido.


### CU03: Registro de capacitaciones y certificaciones

- **Actores involucrados**
  - Personal administrativo y Tripulación
  - Sistema TMS (Transportation Management System)

- **Obejtivo**
  - Garantizar que cada tripulante cuente con cursos y certificaciones vigentes.

- **Precondiciones**
  - El tripulante debe estar registrado en el sistema.

- **Disparador o evento inicial**
  - El tripulante requiere registrar una nueva certificación o renovar una existente para poder embarcar.

- **Flujo principal de eventos**
    1. El personal o tripulante accede al sistema y selecciona la opción “Subir certificación”.
    2. El sistema valida que los datos estén completos y en formato permitido.
    3. El administrador accede al módulo de gestión de personal.
    4. El sistema muestra la lista de certificaciones pendientes de validación.
    5. Si los datos son correctos, el administrador aprueba la certificación y el sistema la marca como “Vigente”.
    6. El sistema vincula la certificación al perfil del tripulante y habilita su disponibilidad para embarque.

- **Flujos alternativos**
    - Si el administrador rechaza la certificación → el sistema notifica al tripulante con el motivo del rechazo.

- **Postcondiciones**
    - La certificación vigente queda asociada al perfil del tripulante.
    - El sistema genera alertas para vencimiento.

- **Excepciones**
    - Intento de registrar una certificación vencida.

### CU04: Administración de esquemas de trabajo y descanso de la tripulación

- **Actores involucrados**
  - Administrador de tripulación (RRHH / Operaciones)
  - Sistema TMS (Transportation Management System)

- **Obejtivo**
  - Definir, controlar y gestionar los periodos de trabajo y descanso de la tripulación.

- **Precondiciones**
  - El tripulante debe estar registrado en el sistema con sus datos completos.
  - El tripulante debe contar con contrato laboral vigente y certificados necesarios.

- **Disparador o evento inicial**
  - El administrador requiere organizar los turnos de embarque y descanso de un tripulante.

- **Flujo principal de eventos**
    1. El administrador accede al módulo de gestión de tripulación.
    2. El sistema muestra la lista de tripulantes activos con sus estados.
    3. El administrador selecciona un tripulante y define el esquema de trabajo y descanso.
    4. El sistema valida que el tripulante tenga contrato vigente y que los periodos no se solapen con otros embarques.
    5. El sistema registra el esquema y actualiza el estado del tripulante.
    6. El sistema programa alertas automáticas cuando se acerque el fin de un periodo para planificar relevos.

- **Flujos alternativos**
    - Si el tripulante no tiene contrato vigente → el sistema alerta y no permite asignar turnos.

- **Postcondiciones**
    - El tripulante queda con un esquema de trabajo/descanso registrado en su historial.
    - El sistema actualiza el estado del trabajador como “contrato vigente”.
    - Se generan alertas para renovación en fechas próximas al vencimiento.

- **Excepciones**
    - Error en la asignación por fallo del sistema.
    - Incompatibilidad con otros embarques previamente registrados.

### CU05: Vincular tripulación con buques específicos y operaciones de embarque

- **Actores involucrados**
  - Administrador de tripulación (RRHH / Operaciones)
  - Tripulante (solo consulta de asignaciones)

- **Obejtivo**
  - Asignar a la tripulación verificada a buques y operaciones de embarque

- **Precondiciones**
  - El tripulante debe contar con contrato laboral vigente.
  - El tripulante debe estar registrado en el sistema con datos completos.
  - El tripulante debe estar registrado en el sistema con sus certificados vigentes.

- **Disparador o evento inicial**
  - El administrador necesita conformar la tripulación para un buque u operación de embarque.

- **Flujo principal de eventos**
    1. El administrador accede al módulo de gestión de tripulación.
    2. El sistema muestra la lista de buques y operaciones de embarque activas.
    3. El administrador selecciona un buque y la operación correspondiente.
    4. El administrador selecciona tripulantes disponibles y los asigna a los roles requeridos.
    5. El sistema valida que los tripulantes tengan contrato y certificaciones vigentes.
    6. El sistema guarda la asignación y actualiza el estado del tripulante como “en embarque”.

- **Flujos alternativos**
    - Si un tripulante no cuenta con certificación vigente → el sistema alerta y no permite la asignación.
    - Si el rol ya está ocupado por otro tripulante → el sistema alerta y solicita corrección.

- **Postcondiciones**
    - La tripulación queda vinculada formalmente al buque y la operación de embarque.
    - El estado de disponibilidad de los tripulantes se actualiza en el sistema.

- **Excepciones**
    - Datos incompletos en la ficha del tripulante o del buque.
    - Intento de asignar un tripulante con contrato o certificación vencida.
