# 11.2.5. Módulo 5 / Integrante 5 

# Módulo de Monitoreo de Entregas

> Proceso Batch `PB_MON_CIERRE_DIARIO_OPERACIONES`

## 1. Objetivo del proceso

El proceso batch `PB_MON_CIERRE_DIARIO_OPERACIONES` tiene como objetivo **consolidar diariamente información de monitoreo de las operaciones terrestres y marítimas** del sistema, generando datos **pre-calculados** en un esquema analítico separado.

A partir de las tablas operacionales de los esquemas `shared`, `monitoreo` y `operaciones_terrestres`, el proceso calcula métricas por operación (duración, incidencias asociadas, medio de transporte, etc.) y carga estos indicadores en una **tabla de hechos** en el schema analítico `monitoreo_analytics`. Este esquema podrá ser utilizado para **consultas analíticas y generación de reportes gerenciales**, siguiendo la idea de un esquema estrella.

## 2. Esquemas y tablas involucradas

### 2.1. Esquemas origen

- **`shared`**
  - `shared.Operacion`: operaciones base del sistema.
  - `shared.OperacionMaritima`: especialización marítima de las operaciones (buque, cantidad de contenedores, porcentaje de trayecto).
  - `shared.OperacionTerrestre`: especialización terrestre de las operaciones.
  - `shared.EstadoOperacion`: catálogo de estados de operación.
  - `shared.Incidencia`: incidencias asociadas a operaciones (tipo, severidad, fecha/hora).

- **`operaciones_terrestres`**
  - `operaciones_terrestres.OperacionTerrestreDetalle`: detalle de operaciones terrestres (vehículo asociado, entre otros datos).

- **`monitoreo`** (opcional, según nivel de detalle)
  - `monitoreo.PosicionBuque`: posiciones GPS de buques.
  - `monitoreo.PosicionVehiculo`: posiciones GPS de vehículos.

### 2.2. Esquema destino (analítico)

Se define un esquema analítico separado, por ejemplo:

- `CREATE SCHEMA monitoreo_analytics;`

Dentro de este esquema se modela un **esquema estrella simplificado**:

- **Tabla de hechos:**

  - `monitoreo_analytics.FactOperacionMonitoreo`  
    Contiene una fila por operación y fecha de corte (normalmente una fila por operación al finalizar).

    Campos sugeridos:
    - `id_fact` (PK, UUID)
    - `id_operacion` (FK lógica a `shared.Operacion`)
    - `id_fecha_corte` (FK a `DimFecha`)
    - `id_estado_operacion` (FK a `DimEstadoOperacion`)
    - `id_buque` (FK a `DimBuque`, si aplica)
    - `id_vehiculo` (FK a `DimVehiculo`, si aplica)
    - `medio_transporte` (`'Marítimo'` / `'Terrestre'` / `'Otro'`)
    - `duracion_minutos`
    - `incidencias_total`
    - `incidencias_alta_severidad`
    - `entregas_asociadas` (si el diseño lo requiere)
    - `fecha_registro_batch` (timestamp de ejecución del batch)

- **Tablas de dimensiones (ejemplo):**

  - `monitoreo_analytics.DimFecha`
    - `id_fecha` (PK, tipo entero o date)
    - `fecha`
    - `anio`, `mes`, `dia`, `nombre_mes`, etc.

  - `monitoreo_analytics.DimBuque`
    - `id_buque` (PK, UUID)
    - `matricula`, `nombre`, `capacidad`, etc. (copiados desde `shared.Buque`)

  - `monitoreo_analytics.DimVehiculo`
    - `id_vehiculo` (PK, UUID)
    - `placa`, `capacidad_ton`, tipo de vehículo, etc. (copiados desde `shared.Vehiculo`)

  - `monitoreo_analytics.DimEstadoOperacion`
    - `id_estado_operacion` (PK, UUID)
    - `nombre` (Programada, En Curso, Completada, Cancelada, etc.)

## 3. Datos de entrada y salida

### 3.1. Entradas

- **Parámetro de entrada del proceso:**
  - `p_fecha_corte DATE`: fecha de corte del cierre diario (normalmente el día calendario que se está cerrando).

- **Datos leídos desde tablas operacionales:**
  - Operaciones finalizadas en `p_fecha_corte`.
  - Información del medio de transporte (buque o vehículo).
  - Incidencias asociadas a cada operación, incluyendo severidad.
  - (Opcional) Últimas posiciones GPS de buque o vehículo para enriquecer el análisis.

### 3.2. Salidas

- **Tabla de hechos:**  
  Inserción de uno o más registros en `monitoreo_analytics.FactOperacionMonitoreo`, con métricas pre-calculadas para cada operación finalizada.

- **Tablas de dimensiones (si se mantiene por batch):**
  - Inserción/actualización de filas en `DimFecha`, `DimBuque`, `DimVehiculo`, `DimEstadoOperacion`, en caso de que aparezcan nuevos valores.

## 4. Descripción del algoritmo del proceso

1. **Determinación del conjunto de operaciones a procesar**
   - Se seleccionan desde `shared.Operacion` todas las operaciones cuya `fecha_fin` corresponda a la `p_fecha_corte` y cuyo estado sea “Completada” (tabla `shared.EstadoOperacion`).

2. **Recorrido secuencial con cursor**
   - Se define un **cursor** sobre el conjunto resultante de operaciones.
   - Se recorre cada operación en un bucle `LOOP ... FETCH ...` en PL/pgSQL.

3. **Cálculo de métricas por operación**
   Para cada operación `o` leída del cursor:

   - **Duración de la operación:**
     - `duracion_minutos = EXTRACT(EPOCH FROM (o.fecha_fin - o.fecha_inicio)) / 60`.

   - **Determinación del medio de transporte:**
     - Si existe registro en `shared.OperacionMaritima` para `o.id_operacion`:
       - Medio = “Marítimo”.
       - Se obtiene `id_buque`.
     - Si existe registro en `shared.OperacionTerrestre` y su detalle en `operaciones_terrestres.OperacionTerrestreDetalle`:
       - Medio = “Terrestre”.
       - Se obtiene `id_vehiculo`.

   - **Cálculo de incidencias asociadas:**
     - Conteo de incidencias totales en `shared.Incidencia` donde `id_operacion = o.id_operacion`.
     - Conteo de incidencias de alta severidad (por ejemplo `grado_severidad >= 4`).

   - **Última posición GPS**:
     - Para operaciones marítimas, se podría registrar la última posición de `monitoreo.PosicionBuque` para el `id_buque`.
     - Para operaciones terrestres, la última posición de `monitoreo.PosicionVehiculo` para el `id_vehiculo`.

4. **Inserción de datos en la tabla de hechos**
   - Se construye un registro de la tabla `monitoreo_analytics.FactOperacionMonitoreo` con:
     - Claves de dimensiones (`id_fecha_corte`, `id_estado_operacion`, `id_buque` o `id_vehiculo`).
     - Métricas calculadas (`duracion_minutos`, `incidencias_total`, `incidencias_alta_severidad`, etc.).
     - `fecha_registro_batch` con la fecha y hora de ejecución del proceso.

5. **Prevención de duplicados**
   - Antes de insertar, el proceso verifica que no exista ya un registro en `FactOperacionMonitoreo` para la combinación:
     - `id_operacion` + `id_fecha_corte`.
   - Si existe, puede:
     - O bien omitir la inserción.
     - O bien actualizar el registro existente (según la política de la solución).

6. **Finalización**
   - Se cierra el cursor.
   - Se deja traza de ejecución (por ejemplo en logs de la aplicación o tabla técnica de auditoría, si se requiere).

## 5. Implementación en PL/pgSQL

A continuación se muestra la función en PL/pgSQL que implementa el proceso batch en el esquema `monitoreo_analytics`. Esta versión incluye el cálculo de la **última posición GPS** (buque o vehículo) y la prevención de duplicados.

```sql
CREATE OR REPLACE FUNCTION monitoreo_analytics.f_cierre_diario_operaciones(p_fecha_corte DATE)
RETURNS void AS
$$
DECLARE
    cur_op CURSOR FOR
        SELECT o.id_operacion,
               o.codigo,
               o.fecha_inicio,
               o.fecha_fin,
               o.id_estado_operacion
        FROM shared.Operacion o
        JOIN shared.EstadoOperacion eo ON eo.id_estado_operacion = o.id_estado_operacion
        WHERE DATE(o.fecha_fin) = p_fecha_corte
          AND eo.nombre = 'Completada';

    v_op RECORD;
    v_id_buque UUID;
    v_id_vehiculo UUID;
    v_duracion_minutos INT;
    v_incidencias_total INT;
    v_incidencias_alta INT;
    v_medio_transporte VARCHAR(20);
    v_id_fecha INT;
    v_ultima_lat DECIMAL(9,6);
    v_ultima_lon DECIMAL(9,6);
    v_fecha_ultima_pos TIMESTAMP;
BEGIN
    -- Asegurar registro en DimFecha
    SELECT id_fecha
    INTO v_id_fecha
    FROM monitoreo_analytics.DimFecha
    WHERE fecha = p_fecha_corte;

    IF NOT FOUND THEN
        INSERT INTO monitoreo_analytics.DimFecha(fecha, anio, mes, dia, nombre_mes, nombre_dia)
        VALUES (
            p_fecha_corte,
            EXTRACT(YEAR FROM p_fecha_corte),
            EXTRACT(MONTH FROM p_fecha_corte),
            EXTRACT(DAY FROM p_fecha_corte),
            TO_CHAR(p_fecha_corte, 'TMMonth'),
            TO_CHAR(p_fecha_corte, 'TMDay')
        )
        RETURNING id_fecha INTO v_id_fecha;
    END IF;

    -- Poblar dimensiones de estado, buque y vehículo a partir de shared
    INSERT INTO monitoreo_analytics.DimEstadoOperacion (id_estado_operacion, nombre)
    SELECT eo.id_estado_operacion, eo.nombre
    FROM shared.EstadoOperacion eo
    ON CONFLICT (id_estado_operacion) DO NOTHING;

    INSERT INTO monitoreo_analytics.DimBuque (id_buque, matricula, nombre, capacidad, peso)
    SELECT b.id_buque, b.matricula, b.nombre, b.capacidad, b.peso
    FROM shared.Buque b
    ON CONFLICT (id_buque) DO NOTHING;

    INSERT INTO monitoreo_analytics.DimVehiculo (id_vehiculo, placa, capacidad_ton)
    SELECT v.id_vehiculo, v.placa, v.capacidad_ton
    FROM shared.Vehiculo v
    ON CONFLICT (id_vehiculo) DO NOTHING;

    OPEN cur_op;

    LOOP
        FETCH cur_op INTO v_op;
        EXIT WHEN NOT FOUND;

        -- Duración en minutos
        v_duracion_minutos := NULL;
        IF v_op.fecha_inicio IS NOT NULL AND v_op.fecha_fin IS NOT NULL THEN
            v_duracion_minutos := (EXTRACT(EPOCH FROM (v_op.fecha_fin - v_op.fecha_inicio)) / 60)::INT;
        END IF;

        -- Determinar medio y activo asociado
        v_id_buque := NULL;
        v_id_vehiculo := NULL;
        v_medio_transporte := 'No definido';

        SELECT om.id_buque
        INTO v_id_buque
        FROM shared.OperacionMaritima om
        WHERE om.id_operacion = v_op.id_operacion;

        IF FOUND THEN
            v_medio_transporte := 'Marítimo';
        ELSE
            SELECT d.id_vehiculo
            INTO v_id_vehiculo
            FROM shared.OperacionTerrestre ot
            JOIN operaciones_terrestres.OperacionTerrestreDetalle d
              ON ot.id_operacion_terrestre = d.id_operacion_terrestre
            WHERE ot.id_operacion = v_op.id_operacion;

            IF FOUND THEN
                v_medio_transporte := 'Terrestre';
            END IF;
        END IF;

        -- Incidencias asociadas
        SELECT COUNT(*)
        INTO v_incidencias_total
        FROM shared.Incidencia i
        WHERE i.id_operacion = v_op.id_operacion;

        SELECT COUNT(*)
        INTO v_incidencias_alta
        FROM shared.Incidencia i
        WHERE i.id_operacion = v_op.id_operacion
          AND i.grado_severidad >= 4;

        -- Última posición GPS según medio de transporte
        v_ultima_lat := NULL;
        v_ultima_lon := NULL;
        v_fecha_ultima_pos := NULL;

        IF v_id_buque IS NOT NULL THEN
            SELECT pb.latitud, pb.longitud, pb.fecha_hora
            INTO v_ultima_lat, v_ultima_lon, v_fecha_ultima_pos
            FROM monitoreo.PosicionBuque pb
            WHERE pb.id_buque = v_id_buque
            ORDER BY pb.fecha_hora DESC
            LIMIT 1;
        ELSIF v_id_vehiculo IS NOT NULL THEN
            SELECT pv.latitud, pv.longitud, pv.fecha_hora
            INTO v_ultima_lat, v_ultima_lon, v_fecha_ultima_pos
            FROM monitoreo.PosicionVehiculo pv
            WHERE pv.id_vehiculo = v_id_vehiculo
            ORDER BY pv.fecha_hora DESC
            LIMIT 1;
        END IF;

        -- Insertar en la tabla de hechos (evitando duplicados)
        IF NOT EXISTS (
            SELECT 1
            FROM monitoreo_analytics.FactOperacionMonitoreo f
            WHERE f.id_operacion = v_op.id_operacion
              AND f.id_fecha_corte = v_id_fecha
        ) THEN
            INSERT INTO monitoreo_analytics.FactOperacionMonitoreo (
                id_operacion,
                id_fecha_corte,
                id_estado_operacion,
                id_buque,
                id_vehiculo,
                medio_transporte,
                duracion_minutos,
                incidencias_total,
                incidencias_alta_severidad,
                ultima_latitud,
                ultima_longitud,
                fecha_ultima_posicion,
                fecha_registro_batch
            ) VALUES (
                v_op.id_operacion,
                v_id_fecha,
                v_op.id_estado_operacion,
                v_id_buque,
                v_id_vehiculo,
                v_medio_transporte,
                v_duracion_minutos,
                v_incidencias_total,
                v_incidencias_alta,
                v_ultima_lat,
                v_ultima_lon,
                v_fecha_ultima_pos,
                NOW()
            );
        END IF;
    END LOOP;

    CLOSE cur_op;
END;
$$ LANGUAGE plpgsql;
```

## 6. Integración con la aplicación

El proceso batch se encuentra completamente integrado con el módulo de monitoreo (backend NestJS y frontend Next.js):

- **Endpoint para ejecutar el batch**
  - Método: `POST`
  - Ruta: `/monitoreo/reportes/cierre-diario`
  - Cuerpo JSON: `{ "fecha_corte": "YYYY-MM-DD" }` (opcional; si no se envía, se usa la fecha actual).
  - Implementación: el servicio `ReportesService` invoca la función PL/pgSQL mediante:
    - `SELECT monitoreo_analytics.f_cierre_diario_operaciones($1)`
  - Este endpoint se expone en la pantalla de **Gestión de Reportes** del frontend, donde el usuario puede seleccionar la fecha de corte y ejecutar manualmente el cierre diario.

- **Endpoint para consultar los datos analíticos**
  - Método: `GET`
  - Ruta: `/monitoreo/reportes/analytics/resumen`
  - Datos que retorna (a partir de `FactOperacionMonitoreo` y dimensiones):
    - KPIs globales: total de operaciones analizadas, duración promedio global, rango de fechas procesadas.
    - Duración promedio por medio de transporte y mes.
    - Incidencias de alta severidad por buque (Top 10).
    - Serie de tiempo por fecha y medio de transporte (duración promedio e incidencias).

- **Pantalla de analítica en el frontend**
  - Ruta: `/monitoreo/reportes/analytics`.
  - Consumidor del endpoint `GET /monitoreo/reportes/analytics/resumen`.
  - Muestra:
    - **KPIs** en tarjetas (operaciones analizadas, duración promedio global, rango de fechas), enriquecidas con iconografía/emoticones.
    - **Gráfico de líneas** (serie de tiempo) de duración promedio por medio de transporte en función de la fecha.
    - **Gráfico de barras** de duración promedio por medio y mes.
    - **Gráfico de barras** de incidencias de alta severidad por buque (Top 10).
  - La navegación se realiza desde la pantalla principal de reportes mediante un botón **"Analítica"**, junto al botón de **"Nuevo Reporte"**.

### 6.1. Orquestación del proceso batch

Actualmente, el proceso `PB_MON_CIERRE_DIARIO_OPERACIONES` se **ejecuta manualmente** desde la pantalla de Gestión de Reportes, donde el usuario selecciona la `fecha_corte` y presiona el botón **"Ejecutar cierre diario"**. Conceptualmente, el proceso está diseñado como un **cierre diario**, pero su disparo puede automatizarse de distintas formas:

- **Opción A: Job en la base de datos (PostgreSQL)**
  - Configurar un job (por ejemplo con `pg_cron`) que se ejecute todos los días a una hora determinada y llame directamente a la función PL/pgSQL:
    - `SELECT monitoreo_analytics.f_cierre_diario_operaciones(CURRENT_DATE - 1);`
  - Ventaja: la ejecución queda cerca de los datos y no depende del backend.

- **Opción B: Job en el backend (NestJS)**
  - Utilizar un scheduler en el backend (por ejemplo `@nestjs/schedule`) para definir una tarea `@Cron` diaria que invoque internamente el método de servicio que hoy llama a la función PL/pgSQL (o el propio endpoint `/monitoreo/reportes/cierre-diario`).
  - Ventaja: la lógica de orquestación queda dentro del mismo módulo de monitoreo.

- **Opción C: Workflow externo (GitHub Actions u otro orquestador)**
  - En un escenario desplegado (backend accesible vía HTTP), definir un workflow con disparador `schedule` que, una vez al día, realice una petición `POST` al endpoint `/monitoreo/reportes/cierre-diario` pasando la `fecha_corte` adecuada.
  - Ventaja: integra la automatización del batch dentro del flujo de DevOps/operación continua.

> Proceso batch `PB_MON_CIERRE_RANGO_FECHAS`

Además del cierre diario, se implementó un proceso batch complementario que permite **consolidar datos analíticos para un rango de fechas de manera masiva**. Esto resulta útil para poblar rápidamente el histórico inicial (por ejemplo, los últimos 120 días del piloto) antes de operar solo con el cierre diario.

### 1. Objetivo del proceso

El proceso `PB_MON_CIERRE_RANGO_FECHAS` (implementado como función `monitoreo_analytics.f_cierre_rango_fechas`) tiene como objetivo:

- Ejecutar el cierre diario de operaciones monitoreadas **para cada día dentro de un rango `[p_fecha_inicio, p_fecha_fin]`**.
- Evitar la ejecución manual día por día cuando se desea cargar un período histórico amplio (por ejemplo, 3 meses de datos generados con scripts de poblamiento).
- Mantener la misma lógica de cálculo y prevención de duplicados que el proceso `PB_MON_CIERRE_DIARIO_OPERACIONES`.

### 2. Datos de entrada y salida

- **Parámetros de entrada:**
  - `p_fecha_inicio DATE`: fecha inicial del rango a procesar.
  - `p_fecha_fin DATE`: fecha final del rango a procesar.

- **Validaciones básicas:**
  - Ninguno de los parámetros puede ser nulo.
  - `p_fecha_inicio` no puede ser mayor que `p_fecha_fin`.

- **Salidas:**
  - Inserción de múltiples registros en `monitoreo_analytics.FactOperacionMonitoreo` para todas las operaciones completadas en cada día del rango.
  - Reutiliza las tablas de dimensiones y la lógica de prevención de duplicados ya descrita para el cierre diario.

### 3. Descripción del algoritmo

El algoritmo de `f_cierre_rango_fechas` es simple y se apoya completamente en la función diaria `f_cierre_diario_operaciones`:

1. Validar que `p_fecha_inicio` y `p_fecha_fin` no sean nulos y que `p_fecha_inicio <= p_fecha_fin`.
2. Inicializar una variable de trabajo `v_fecha := p_fecha_inicio`.
3. Mientras `v_fecha <= p_fecha_fin`:
   - Invocar `monitoreo_analytics.f_cierre_diario_operaciones(v_fecha)`.
   - Avanzar `v_fecha := v_fecha + INTERVAL '1 day'`.
4. Finalizar cuando se alcance la `p_fecha_fin`.

Como la función diaria ya implementa la verificación de **duplicados por combinación (`id_operacion`, `id_fecha_corte`)**, el proceso de rango es **idempotente por día**. Es decir, si algún día del rango ya fue procesado previamente por el cierre diario, la función de rango no insertará filas duplicadas para ese día.

### 4. Implementación en PL/pgSQL

La función se encuentra definida en el mismo script de DDL consolidado del módulo:

```sql
CREATE OR REPLACE FUNCTION monitoreo_analytics.f_cierre_rango_fechas(
  p_fecha_inicio DATE,
  p_fecha_fin    DATE
)
RETURNS void AS
$$
DECLARE
  v_fecha DATE;
BEGIN
  IF p_fecha_inicio IS NULL OR p_fecha_fin IS NULL THEN
    RAISE EXCEPTION 'Las fechas no pueden ser nulas';
  END IF;

  IF p_fecha_inicio > p_fecha_fin THEN
    RAISE EXCEPTION 'p_fecha_inicio (%) no puede ser mayor que p_fecha_fin (%)',
      p_fecha_inicio, p_fecha_fin;
  END IF;

  v_fecha := p_fecha_inicio;
  WHILE v_fecha <= p_fecha_fin LOOP
    PERFORM monitoreo_analytics.f_cierre_diario_operaciones(v_fecha);
    v_fecha := v_fecha + INTERVAL '1 day';
  END LOOP;
END;
$$ LANGUAGE plpgsql;
```

### 5. Integración con la aplicación (cierre masivo de 120 días)

Para facilitar su uso durante el piloto, se expuso un endpoint específico en el backend NestJS que llama a `f_cierre_rango_fechas` para un rango fijo de **120 días**:

- **Endpoint para ejecutar el cierre masivo de 120 días**
  - Método: `POST`
  - Ruta: `/monitoreo/reportes/cierre-rango-120`
  - Cuerpo JSON: `{ "fecha_fin": "YYYY-MM-DD" }` (opcional; si no se envía, se usa la fecha actual del sistema).
  - Lógica:
    - Determina `fecha_fin` a partir del parámetro o del día actual.
    - Calcula `fecha_inicio = fecha_fin - 119 días` (obteniendo un total de 120 días calendario).
    - Invoca `SELECT monitoreo_analytics.f_cierre_rango_fechas(fecha_inicio, fecha_fin);`.

- **Pantalla de Gestión de Reportes**
  - En la pantalla `/monitoreo/reportes` se muestra un bloque de **“Proceso batch analítico”** con:
    - Un selector de fecha (`Fecha de corte`).
    - Un botón para **"Ejecutar cierre diario"** (llama a `/monitoreo/reportes/cierre-diario`).
    - Un botón adicional para **"Cierre últimos 120 días"** (llama a `/monitoreo/reportes/cierre-rango-120`).
  - Este segundo botón se utiliza principalmente para **poblar el histórico inicial del piloto** (por ejemplo, los últimos 3 meses generados con los scripts de datos), mientras que el botón de cierre diario se usa para el mantenimiento operativo día a día.

- **Página de analítica**
  - La página `/monitoreo/reportes/analytics` consume todos los datos consolidados en `FactOperacionMonitoreo`, **sin distinguir** si fueron generados por el cierre diario o por el cierre masivo de 120 días.
  - Se incluye una nota informativa en la propia pantalla indicando la recomendación de:
    - Ejecutar **una sola vez** el cierre de los últimos 120 días para poblar el histórico.
    - Utilizar luego **solo el cierre diario** para mantener la información actualizada.
