# 6.1. Módulo / Integrante 1 – Gestión de Reservas

## Tabla: Cliente

### **1. Justificación de la selección**

La tabla **Cliente** fue seleccionada para el análisis físico por las siguientes razones:

- **Relevancia transaccional:** Es la entidad central que identifica a los clientes que generan reservas.  
- **Criticidad:** Varios procesos dependen de ella (contacto, historial).  

---

### **2. Estimación de cantidad de registros**

Se usan los supuestos proporcionados: año base **1,000 clientes**.  
Crecimientos: **+7%** (Año 2) y **+8%** (Año 3, sobre Año 2).

#### **Supuestos**

- Año base: 1,000 clientes.  
- Crecimiento anual:
  - Año 2 → +7%
  - Año 3 → +8% (aplicado sobre Año 2)

#### **Proyección**

| Año | Nuevos Clientes | Total Clientes | Crecimiento |
|-----|------------------|----------------|--------------|
| 1 | 1,000 | 1,000 | — |
| 2 | 70 | 1,070 | 7% |
| 3 | 86 | 1,156 | 8% |

---

### **3. Script SQL de creación de tabla**

```sql
CREATE TABLE MODULO_RESERVAS.Cliente (
    ruc CHAR(11) PRIMARY KEY,
    razon_social VARCHAR(150) NOT NULL,
    direccion VARCHAR(200) NOT NULL,
    email VARCHAR(100),
    CONSTRAINT chk_ruc_cliente CHECK (ruc ~ '^[0-9]{11}$'),
    CONSTRAINT chk_email_cliente CHECK (email IS NULL OR email ~* '^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$')
);

CREATE INDEX idx_razon_social ON MODULO_RESERVAS.Cliente(razon_social);
```

---

### **4. Estructura del registro**

| Atributo                         | Formato      | Tamaño        |
| -------------------------------- | ------------ | ------------- |
| ruc                              | CHAR(11)     | 11 bytes      |
| razon_social                     | VARCHAR(150) | 150 bytes     |
| direccion                        | VARCHAR(200) | 200 bytes     |
| email                            | VARCHAR(100) | 100 bytes     |
| **Total estimado por registro:** |              | **461 bytes** |

---

### **5. Resumen de parámetros físicos**

| Parámetro                         | Valor                                                        |
| --------------------------------- | ------------------------------------------------------------ |
| Tamaño de bloque                  | 8 KB (8,192 bytes)                                           |
| Tamaño de extent                  | ---                                                          |
| Estructura del registro lógico    | 461 bytes                                                    |
| Overhead de fila                  | 23 bytes                                                     |
| Filas por bloque (estimado)       | 13                                                           |
| Espacio libre reservado           | `ALTER TABLE MODULO_RESERVAS.Cliente SET (FILLFACTOR = 80);` |
| **Tamaño total estimado (año 3)** | **≈ 729,088 bytes (≈ 712 KB)**                               |

---

### **6. Cálculos**

**Espacio libre por bloque:**  
(8,192 − 24) × 0.8 = **6,534.4 bytes por bloque**

**Tamaño por fila con overhead:**  
461 + 23 = **484 bytes**

**Filas por bloque:**  
6,534.4 ÷ 484 ≈ 13.51 ≈ **13 filas por bloque**

**Cantidad de filas (año 3):**  
**1,156 filas**

**Cantidad de bloques necesarios:**  
1,156 ÷ 13 ≈ 88.92 ≈ **89 bloques**

**Tamaño total estimado:**  
89 × 8,192 = **729,088 bytes ≈ 712 KB**

## Tabla: Reserva

### **1. Justificación de la selección**

La tabla Reserva fue seleccionada para el análisis físico por las siguientes razones:

**Volumen transaccional:** Es la tabla principal de transacciones del módulo. Se esperan inserciones constantes por cada reserva efectuada.

**Criticidad operativa:** Define asignaciones a buque, ruta y agente. Impacta procesos de planificación y facturación.

**Relevancia para rendimiento:** Consultas frecuentes por fecha, cliente, estado y agente exigen diseño físico optimizado.

---

### **2. Estimación de cantidad de registros**

Año base 4,000 reservas. Crecimientos: +7% (Año 2) y +8% (Año 3 sobre Año 2).

**Supuestos:**
- **Año base:** 4,000 reservas
- **Crecimiento anual:**
  - Año 2 → +7%
  - Año 3 → +8% (aplicado sobre Año 2)

**Proyección:**
| Año | Nuevas Reservas | Total Reservas | Crecimiento |
|-----|-----------------|----------------|-------------|
| 1   | 4,000           | 4,000          | ---         |
| 2   | 280             | 4,280          | 7%          |
| 3   | 342             | 4,622          | 8%          |

---

### **3. Script SQL de creación de tabla**

```sql
CREATE TABLE MODULO_RESERVAS.Reserva (
    id_reserva INT PRIMARY KEY,
    codigo VARCHAR(20) NOT NULL UNIQUE,
    fecha_registro DATE NOT NULL DEFAULT CURRENT_DATE,
    id_estado_reserva INT NOT NULL,
    pago_total DECIMAL(10,2),
    ruc_cliente CHAR(11) NOT NULL,
    id_agente_reservas INT NOT NULL,
    matricula_buque VARCHAR(20) NOT NULL,
    id_ruta INT NOT NULL,
    CONSTRAINT fk_reserva_estado FOREIGN KEY (id_estado_reserva)
        REFERENCES MODULO_RESERVAS.EstadoReserva(id_estado_reserva)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    CONSTRAINT fk_reserva_cliente FOREIGN KEY (ruc_cliente)
        REFERENCES MODULO_RESERVAS.Cliente(ruc)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    CONSTRAINT fk_reserva_agente FOREIGN KEY (id_agente_reservas)
        REFERENCES MODULO_RESERVAS.AgenteReservas(id_agente_reservas)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    CONSTRAINT fk_reserva_buque FOREIGN KEY (matricula_buque)
        REFERENCES MODULO_RESERVAS.Buque(matricula)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    CONSTRAINT fk_reserva_ruta FOREIGN KEY (id_ruta)
        REFERENCES MODULO_RESERVAS.Ruta(id_ruta)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    CONSTRAINT chk_pago_total CHECK (pago_total IS NULL OR pago_total >= 0)
);

CREATE INDEX idx_reserva_cliente ON MODULO_RESERVAS.Reserva(ruc_cliente);
CREATE INDEX idx_reserva_estado ON MODULO_RESERVAS.Reserva(id_estado_reserva);
CREATE INDEX idx_reserva_fecha ON MODULO_RESERVAS.Reserva(fecha_registro);
```

---

### **5. Resumen de parámetros físicos**

| Parámetro                         | Valor                                                        |
| --------------------------------- | ------------------------------------------------------------ |

### **6. Cálculos**

**Espacio libre por bloque:**  
(8,192 − 24) × 0.8 = **6,534.4 bytes por bloque**

**Tamaño por fila con overhead:**  
79 + 23 = **102 bytes**

**Filas por bloque:**  
6,534.4 ÷ 102 ≈ 64.06 ≈ **64 filas por bloque**

**Cantidad de filas (año 3):**  
**4,622 filas**

**Cantidad de bloques necesarios:**  
4,622 ÷ 64 ≈ 72.22 ≈ **73 bloques**

**Tamaño total estimado:**  
73 × 8,192 = **598,016 bytes ≈ 584 KB**
